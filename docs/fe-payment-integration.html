Veggieverse Store 결제 — Frontend 통합 가이드

토스 페이먼츠 단건 결제 플로우를 Veggieverse Store에 통합하기 위한 프론트엔드 개발 가이드입니다. 백엔드(PR3·PR4)는 구현 완료되어 있으며, 본 문서는 FE가 구현해야 할 범위와 BE API 계약을 정의합니다.

2. Step 1 — PENDING 주문 생성 (BE 호출)

사용자가 결제하기 버튼을 누르면 가장 먼저 BE에 주문을 등록하고 토스에 보낼 식별자(tossOrderId)와 금액을 받아옵니다. 토스 SDK 호출에 필요한 입력값을 모두 이 응답에서 얻습니다.

요청

MethodPOST
URL/api/v1/veggieverse/store/orders
AuthAuthorization: Bearer <JWT> REQUIRED
Content-Typeapplication/json

요청 바디 필드

필드타입필수설명
items Array<{productId: number, quantity: number}> 필수 비어 있을 수 없음. productId 중복 금지.
deliveryAddress {zipCode?, street?, detail?} 필수 객체 자체는 필수. 내부 필드는 모두 선택.
isCartOrder boolean 필수 장바구니에서 시작한 주문인지 여부. true면 confirm 성공 시점에 BE가 cart를 자동으로 비웁니다. 즉시 구매(상품 상세에서 바로 결제)는 false.
couponId string 선택 현 단계(PR3)에서는 BE가 받아만 두고 가격에 반영하지 않습니다. 추후 확정.

요청 예시

POST /api/v1/veggieverse/store/orders
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

{
  "items": [
    { "productId": 101, "quantity": 2 },
    { "productId": 207, "quantity": 1 }
  ],
  "deliveryAddress": {
    "zipCode": "04524",
    "street": "서울시 중구 세종대로 110",
    "detail": "5층"
  },
  "isCartOrder": true
}

응답 (201 Created)

필드타입설명Toss SDK 매핑
orderDbId number 우리 DB의 주문 PK. 주문 상세 조회/취소 등에서 사용. —
tossOrderId string 토스에 전달할 주문 식별자. orderNumber와 동일. orderId
orderName string 결제창 상단에 노출되는 주문명 (예: "비건 김치 외 2건"). orderName
amount number (BigDecimal) 최종 결제 금액. Step 4 confirm 호출 시 이 값과 일치해야 함. amount.value
currency string 기본 "KRW". amount.currency
customerEmail string? 영수증·알림용. 있으면 SDK에 그대로 전달. customerEmail
customerName string? 영수증·알림용. customerName
customerMobilePhone string? 영수증·알림용. customerMobilePhone

3. Step 2 — 토스 결제창 띄우기 (SDK 호출)

Step 1 응답을 그대로 들고 Toss JS SDK를 호출하여 결제창을 띄웁니다. 결제창은 Toss가 렌더링하며, FE는 입력에 관여하지 않습니다.

SDK 키

환경clientKey
테스트 test_ck_docs_Ovk5rk1EwkEbP0W43n07xlzm
프로덕션 실제 발급된 키 (배포 환경변수로 주입)

호출 예시 (TypeScript)

import { loadTossPayments } from "@tosspayments/payment-sdk";

const order = await createPendingOrder(...);   // Step 1 응답
const toss = await loadTossPayments(CLIENT_KEY);

await toss.requestPayment("카드", {
  amount: order.amount,
  orderId: order.tossOrderId,                  // ⚠ orderDbId 아님
  orderName: order.orderName,
  customerEmail: order.customerEmail ?? undefined,
  customerName: order.customerName ?? undefined,
  customerMobilePhone: order.customerMobilePhone ?? undefined,
  successUrl: `${window.location.origin}/order/success`,
  failUrl:    `${window.location.origin}/order/fail`,
});
주의 — orderId vs orderDbId: Toss SDK의 orderId에는 반드시 응답의 tossOrderId를 넣으세요. orderDbId(숫자 PK)를 잘못 넣으면 confirm 단계에서 주문을 찾을 수 없습니다.

4. Step 3 — 결제 결과 대기 페이지 (success/fail 라우트)

Toss가 인증 후 사용자를 successUrl 또는 failUrl로 리다이렉트합니다. FE는 두 경로에 해당하는 페이지를 모두 구현해야 합니다.

successUrl — 쿼리 파라미터

파라미터설명
paymentKeyToss가 발급한 결제 식별자. confirm 호출에 필요.
orderIdStep 1의 tossOrderId와 동일.
amount인증된 결제 금액. Step 1의 amount와 동일해야 함.

failUrl — 쿼리 파라미터

파라미터설명
codeToss 에러 코드 (예: USER_CANCEL, PAY_PROCESS_ABORTED).
message사람이 읽을 수 있는 에러 메시지.
orderId해당 주문의 tossOrderId.
successUrl 페이지는 단순한 "로딩 화면"입니다. 페이지가 로드되면 곧바로 Step 4의 confirm API를 호출해야 하며, 사용자에게 "결제 완료"를 보여주는 시점은 confirm API가 200 OK를 반환한 이후입니다.

5. Step 4 — 최종 결제 승인 요청 (BE 호출)

Toss 인증만 끝났을 뿐 실제 청구는 아직 일어나지 않았습니다. successUrl 페이지에서 쿼리 파라미터 3개를 추출하여 BE confirm API를 호출해야 비로소 결제가 확정됩니다.

요청

MethodPOST
URL/api/v1/veggieverse/store/payments/confirm
AuthAuthorization: Bearer <JWT> REQUIRED
Content-Typeapplication/json

요청 바디

{
  "paymentKey": "<successUrl 쿼리의 paymentKey 그대로>",
  "orderId":    "<successUrl 쿼리의 orderId 그대로>",
  "amount":     <successUrl 쿼리의 amount 그대로>
}

응답 (200 OK) — OrderDetailResponse

주문 상세 객체 전체를 반환합니다. 이 응답을 그대로 "결제 완료 화면"에 사용할 수 있어 별도 "주문 상세 조회"를 다시 호출할 필요가 없습니다.

이 응답이 200으로 돌아온 시점부터만 "결제 완료"로 간주합니다. Toss successUrl 도달 ≠ 결제 완료입니다.

6. Step 5 — 예외·실패 처리

사용자가 결제창을 닫거나, Toss가 거절하거나, BE confirm이 실패하는 경로별로 적절한 UI를 제공해야 합니다. 특히 confirm 응답의 HTTP 상태 코드별로 안내·재시도 전략이 달라야 합니다.

6.1 결제창 단계 (Toss SDK)

상황경로FE 처리
사용자가 결제창을 닫음 failUrl 리다이렉트 (code=USER_CANCEL) "결제가 취소되었습니다" 안내, 장바구니로 복귀 옵션 제공.
잔액 부족·카드 거절 failUrl 리다이렉트 (code=...) Toss message를 그대로 노출 + "다른 결제수단으로 재시도" 버튼.
SDK 호출 자체 실패 (네트워크 등) requestPayment Promise reject "결제창을 띄우지 못했습니다" + 페이지 유지(주문은 PENDING, 그대로 재시도 가능).

6.2 confirm API 단계 (BE 응답)

HTTP 의미 발생 원인 FE UX
200 결제 확정 정상 승인 완료 화면으로 전환, 응답 바디로 주문 정보 렌더링.
400 AmountMismatch 금액 변조 의심 쿼리 amount가 Order의 finalAmount와 불일치. 재시도 의미 없음. "주문 정보가 일치하지 않습니다" → 처음 단계로 돌려보냄.
410 OrderExpired PENDING 만료 주문 생성 후 10분 초과. "결제 가능 시간이 만료되었습니다" → 새 주문 생성 유도 (Step 1부터 다시).
400 PaymentRejected Toss 4xx 거절 카드 한도 초과, 잘못된 카드, 미지원 결제수단 등. BE 에러 메시지를 그대로 노출 + "다른 결제수단으로 재시도" 버튼.
503 PaymentRetryable 일시적 오류 Toss 5xx / 네트워크 / 타임아웃. Order는 PENDING 유지. "잠시 후 다시 시도해주세요" + 재시도 버튼(동일 paymentKey/orderId로 같은 confirm을 다시 호출).
503 재시도 시 주의: 503은 BE가 Toss 결과를 알지 못한 상태이며 Order는 PENDING으로 유지됩니다. 사용자가 페이지를 닫더라도 동일한 paymentKey·orderId·amount로 confirm을 다시 호출하면 멱등하게 처리됩니다(새 주문 생성 X).

7. FE 산출물 체크리스트

#산출물핵심 동작
1 결제하기 버튼 핸들러 ① POST /orders 호출 (items, deliveryAddress, isCartOrder 포함) → ② Toss SDK requestPayment 호출 (tossOrderId·amount·orderName·customer* 매핑)
2 결제 성공 라우트 (/order/success) 쿼리 paymentKey·orderId·amount 추출 → POST /payments/confirm 호출 → 200 OK 받으면 완료 화면 표시, 실패 시 §6.2 분기 처리
3 결제 실패 라우트 (/order/fail) 쿼리 code·message·orderId 추출 → 사유 안내 → 재시도 또는 장바구니 복귀
4 완료 화면 confirm 응답(OrderDetailResponse)으로 주문 번호·금액·배송지 렌더링
5 에러 메시지 매핑 테이블 HTTP 400(AmountMismatch / PaymentRejected) · 410 · 503 각각에 대응되는 안내 + 재시도 정책
